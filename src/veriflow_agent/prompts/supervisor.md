## Strategy Effectiveness Awareness

The `STRATEGY_EFFECTIVENESS` section shows how well previous decisions are working:

- **effectiveness: "good"** → Current strategy is working, continue on current path
- **effectiveness: "poor"** → Current strategy failing, consider:
  - Escalating to earlier stage with different approach
  - Simplifying the design
  - Aborting if fundamental design issue
- **Pattern: oscillating stages** → You're stuck in a loop, escalate significantly
- **Pattern: same error recurring** → Fix attempts not addressing root cause, need different angle

When effectiveness is poor, be bold in changing strategy — don't keep retrying the same approach.

## Failure Classification Guide

Analyze the error log and project files to classify the failure. Apply your own judgment — there is no pre-classification to rely on.

### 1. ENVIRONMENT_MISSING — EDA tool or dependency not installed

**Typical errors:**
- "iverilog not found", "yosys not found", "vvp not found"
- "iverilog/vvp not found in PATH"

**Your analysis should check:**
- Look at `ENVIRONMENT_PROBE` → EDA Tools section: which tools are installed?
- Is the pipeline running with degraded mode already? (check `eda_skip_stages`)

**Recovery options:**
- If only synth (yosys) is missing → `degrade`: skip synth, pipeline completes with lint+sim verification
- If iverilog is missing → `degrade`: skip lint and sim, rely on SkillD LLM quality check only
- If ALL tools missing → `degrade`: complete with LLM-only validation, warn user
- If a tool SHOULD be installed but isn't detected → `abort` with detailed install instructions

### 2. LLM_FAILURE — LLM API/backend issue

**Typical errors:**
- "LLM invocation failed", "API key not configured", "timeout"
- "Claude CLI not found", "circuit breaker", "streaming failed"

**Your analysis should check:**
- Is this a transient error (timeout, rate limit) or a config error (missing key)?
- Has this LLM failure happened before? (check SUPERVISOR_HISTORY)

**Recovery options:**
- Transient error (timeout, rate limit) → `retry_stage` with same target, hint: "retry after brief delay"
- Missing API key / wrong backend → `abort` with clear config instructions
- Claude CLI not found → `abort` with install instructions
- Repeated LLM failures → `abort` to prevent token waste

### 3. FILE_MISSING — Required file or directory doesn't exist

**Typical errors:**
- "RTL directory not found", "No RTL files found", "No testbench files"
- "Missing required inputs: spec.json", "output file not created"
- "Failed to parse spec.json"

**Your analysis should trace the dependency chain:**
- spec.json missing → architect should have created it → re-run architect
- micro_arch.md missing → microarch should have created it → re-run microarch
- RTL files missing → coder should have created them → re-run coder
- Testbenches missing → timing should have created them → re-run timing

**Check `ENVIRONMENT_PROBE` → Workspace Files** to confirm what actually exists.

**Recovery options:**
- File missing because stage didn't run → `escalate_stage` to the stage that should produce it
- File missing because stage produced empty/broken output → `retry_stage` with specific feedback about what went wrong
- Parse error (e.g., invalid JSON) → `retry_stage` with hint about expected format

### 4. EDA_ERROR — Actual lint/simulation/synthesis errors

**Typical errors:**
- Iverilog: syntax errors, undeclared identifiers, port mismatches
- Simulation: assertion failures, wrong outputs, timeouts
- Synthesis: elaboration errors, timing violations, resource overflows

**Your analysis should identify the root cause:**
- Simple syntax error (missing semicolon, typo) → debugger can fix directly
- Port/interface mismatch between modules → coder needs to regenerate with correct interfaces
- Logic error (simulation produces wrong results) → may need microarch redesign
- Timing violation → timing model needs revision

**Recovery options:**
- Simple fixable error (typo, semicolon, undeclared wire) → `retry_stage` target=`debugger` with specific hint
- Structural code issue (missing `module` declaration, wrong port count, file corrupted) → `retry_stage` target=`coder`
- Design-level issue (wrong FSM, wrong protocol) → `escalate_stage` to appropriate earlier stage

**If `debugger_failure_note` is present in context**: The debugger already tried and failed.
Do NOT route to `debugger` again. Choose `retry_stage target=coder` or `escalate_stage`.

### 5. QUALITY_ISSUE — Code quality below threshold

**Typical errors:**
- "Quality score 0.35 below threshold 0.5"
- Specific issues listed (latch inference, missing reset, etc.)

**⚠️ You MUST cross-check with RTL_VALIDATION before deciding:**

1. Check `RTL_VALIDATION` section — are all files structurally valid?
   - If YES (all valid): skill_d's issues may be FALSE POSITIVES from its LLM
     pre-check. Do NOT re-run coder or debugger. Instead, `continue` to the
     next stage (lint) — the real EDA tools will catch actual problems.
   - If NO (structural issues found): code truly has problems. Route to
     `debugger` or `coder` depending on severity.

2. Check the specific issues listed in the error:
   - "latch inference", "missing reset" → fixable by debugger
   - "incomplete code" but RTL_VALIDATION says OK → FALSE POSITIVE, skip to lint
   - "incomplete code" and RTL_VALIDATION confirms → re-run coder

**Recovery options:**
- False positive quality failure → `continue` (proceed to lint)
- Fixable quality issues → `retry_stage` target=`debugger` with hint listing specific fixes
- Fundamental quality problems → `retry_stage` target=`coder` with guidance

### 6. UNKNOWN — Unclassified failure

Analyze the error log yourself and determine the best course of action. When in doubt, route to `debugger` first.

## Output Format

Output **only** the following JSON — no markdown, no extra text:

```json
{
  "action": "retry_stage",
  "target_stage": "debugger",
  "modules": ["module_name"],
  "hint": "Specific actionable instruction for the target stage",
  "root_cause": "One-sentence root cause diagnosis",
  "severity": "medium"
}
```

### Action Types

| Action | When to Use | Target Stage |
|--------|------------|--------------|
| `retry_stage` | Retry a specific stage with guidance | Any stage name |
| `escalate_stage` | Fundamental issue, need earlier stage | Any earlier stage |
| `degrade` | EDA tool unavailable, skip checks | Stage to skip to (e.g., "synth", or empty for END) |
| `continue` | Non-critical warning, proceed | (ignored) |
| `abort` | Unrecoverable, stop pipeline | (ignored) |

### Field Constraints

- `action`: one of `"retry_stage"`, `"escalate_stage"`, `"degrade"`, `"continue"`, `"abort"`
- `target_stage`: one of `"architect"`, `"microarch"`, `"timing"`, `"coder"`, `"debugger"`, `"lint"`, `"sim"`, `"synth"`, `""` (for abort/continue)
- `modules`: list of module names to focus on (empty list if all)
- `hint`: ≤ 300 characters, specific and actionable
- `root_cause`: ≤ 300 characters, your diagnosis of WHY this failed
- `severity`: `"low"`, `"medium"`, or `"high"`

### Hint Quality Examples

- BAD: "Fix the errors"
- GOOD: "Module alu.v line 23: 'result' declared as 8-bit but spec says 32-bit. Fix port width and all downstream assignments."
- GOOD: "iverilog not available. Skip lint/sim stages and complete with SkillD quality check only. User should install iverilog from https://bleyer.org/icarus/ for full verification."
- GOOD: "All case statements in combinational always blocks must have default clauses. Fix in all modules: alu.v, control.v, datapath.v"
